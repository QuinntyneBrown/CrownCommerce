# L2 - Detailed Requirements

## L2-001: Product Listing Page
**Traces to:** L1-001

The system must provide a shop page that displays all hair products in a grid layout with filtering by category (bundle, wig, closure, frontal), texture (straight, curly, wavy, kinky, body wave), and origin country, plus sorting options. Product data must be fetched from the Catalog API.

**Acceptance Criteria:**
1. Given a customer navigates to the shop page, when the page loads, then all products from `CatalogService.getProducts()` must be displayed as product cards in a responsive grid.
2. Given products are displayed, when the customer selects a category filter, then only products matching that category must be shown.
3. Given products are displayed, when the customer selects a texture filter, then only products matching that texture must be shown.
4. Given products are displayed, when the customer selects an origin filter, then only products from that origin country must be shown.
5. Given products are displayed, when the customer selects a sort option, then products must reorder by the selected criterion (price low-high, price high-low, name A-Z).
6. Given viewport width < 576px, when viewing the shop page, then the product grid must display in a single-column layout.
7. Given viewport width >= 992px, when viewing the shop page, then the product grid must display in a multi-column layout (3-4 columns).

---

## L2-002: Product Detail Page
**Traces to:** L1-001

The system must provide a product detail page accessible via route `/product/:id` that displays full product information including name, description, price, origin, texture, type, length, and image, with an add-to-cart control.

**Acceptance Criteria:**
1. Given a customer navigates to `/product/:id`, when the page loads, then the product details must be fetched from `CatalogService.getProduct(id)` and displayed.
2. Given a product detail page is loaded, when the customer selects a quantity and clicks "Add to Cart", then the item must be added to the cart via `OrderService.addToCart()`.
3. Given an invalid product ID in the URL, when the page loads, then a "Product not found" message or redirect to the 404 page must occur.

## L2-003: Admin Product CRUD
**Traces to:** L1-001

The admin panel must provide full CRUD operations for hair products: listing with search/filter/pagination, a creation form with all product fields bound to the API, edit capability, and delete with confirmation.

**Acceptance Criteria:**
1. Given an admin navigates to the products list page, when the page loads, then products must be fetched from `CatalogService.getProducts()` and displayed in a data table.
2. Given the products list, when the admin types in the search input, then the table must filter products by name.
3. Given the products list, when the admin clicks "Add Product", then a product creation form must open with fields for name, origin (from API), texture, type, length, price, image URL, and description.
4. Given a filled product form, when the admin clicks "Save Product", then the product must be created via `CatalogService.createProduct()` and the admin must be redirected to the products list.
5. Given the products list, when the admin clicks "Edit" on a row, then the form must load pre-populated with that product's data.
6. Given the products list, when the admin clicks "Delete" on a row, then a confirmation dialog must appear, and on confirm, the product must be deleted via `CatalogService.deleteProduct()`.
7. Given more products than fit on one page, when pagination controls are used, then the next/previous page of results must load.

## L2-005: Shopping Cart
**Traces to:** L1-002

The system must provide a cart page that displays all items added to the customer's cart with quantities, individual prices, and total, with the ability to remove items, clear the cart, and navigate to checkout.

**Acceptance Criteria:**
1. Given a customer navigates to `/cart`, when the page loads, then all cart items must be displayed with product name, quantity, unit price, and line total.

## L2-006: Checkout Flow
**Traces to:** L1-003

The system must provide a checkout page with a multi-step form collecting shipping information and payment details, creating an order via `OrderService` and processing payment via `PaymentService`.

**Acceptance Criteria:**
1. Given a customer navigates to `/checkout` with items in cart, when the page loads, then an order form must be displayed with fields for shipping address and payment details.
2. Given a filled checkout form, when the customer submits the order, then an order must be created via `OrderService.createOrder()`.
3. Given an order is created, when payment is initiated, then `PaymentService.createPayment()` must be called and the payment must be confirmed via `PaymentService.confirmPayment()`.
4. Given successful payment, when the process completes, then an order confirmation must be displayed with the order details.
5. Given a validation error in the form, when the customer attempts to submit, then validation errors must be displayed inline next to the invalid fields.

## L2-007: Payment Integration
**Traces to:** L1-004

The Payment microservice must support creating payments, confirming payments, retrieving payment status, and processing refunds.

**Acceptance Criteria:**
1. Given a valid order, when `POST /api/payments` is called with order details, then a payment record must be created and a payment ID returned.
2. Given a pending payment, when `POST /api/payments/{id}/confirm` is called, then the payment status must update to confirmed.
3. Given a confirmed payment, when `POST /api/payments/{id}/refund` is called, then a refund must be processed and the payment status must update accordingly.
4. Given an invalid payment ID, when any payment endpoint is called, then a 404 response must be returned.

---

## L2-008: User Registration and Login
**Traces to:** L1-005

The system must provide registration and login endpoints in the Identity microservice, with the frontend providing login pages in both consumer storefronts and the Teams app.

**Acceptance Criteria:**
1. Given a new user, when `POST /api/identity/auth/register` is called with name, email, password, and role, then a user account must be created and a JWT token returned.
2. Given a registered user, when `POST /api/identity/auth/login` is called with valid credentials, then a JWT token must be returned containing user ID, email, name, and role claims.
3. Given invalid credentials, when login is attempted, then a 401 response must be returned with an error message.
4. Given a valid JWT token, when `GET /api/identity/auth/profile` is called, then the authenticated user's profile must be returned.
5. Given no JWT token, when a protected endpoint is called, then a 401 Unauthorized response must be returned.

---

## L2-009: Auth Guards and Interceptors
**Traces to:** L1-005

All authenticated frontend applications must use an HTTP interceptor to attach JWT tokens to outgoing API requests and auth guards to protect routes requiring authentication.

**Acceptance Criteria:**
1. Given an authenticated user, when any HTTP request is made to the API, then the `authInterceptor` must attach the JWT token as a Bearer Authorization header.
2. Given an unauthenticated user, when they attempt to navigate to a protected route, then the auth guard must redirect them to the login page.
3. Given a user clicks logout, when the action completes, then the JWT token must be cleared and the user must be redirected to the login page.

**Acceptance Criteria:**
1. Given a team member navigates to the Teams app without authentication, when the auth guard triggers, then they must be redirected to `/login`.
2. Given the login page, when the user enters valid credentials and submits, then `AuthService.login()` must be called and on success the user must be redirected to `/home`.
3. Given invalid credentials, when login is attempted, then an error message must be displayed.
4. Given successful login, when the token is stored, then the `authInterceptor` must attach it to all subsequent API requests.
